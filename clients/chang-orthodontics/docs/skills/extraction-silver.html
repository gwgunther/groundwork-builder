Skill: Silver Extraction

Responsibility

The bronze→silver transform. Takes raw scraped HTML/JSON-LD/text from up to 8 key pages of a practice site and extracts a single structured PracticeData JSON object: NAP, doctor(s), services, brand colors, hours, content blocks (hero/about/testimonials/faqs), images-by-role, differentiators (was "signals"), and a verbatim content rescue layer (additionalContent[]). This is the foundation of the entire pipeline — every downstream skill depends on its accuracy.

Inputs

Field Type Source Notes
bronze.baseUrl string scrape Site root URL
bronze.pageCount number scrape Total pages crawled
allPaths (first 40) string[] scrape All known URL paths
pageBlocks string derived Formatted bodies of up to 8 selected key pages (homepage, about, contact, doctor pages, services, team, schedule, specials)
siteAssets.navigation string[] scrape Top-level nav text
siteAssets.socialLinks string[] scrape Up to 8 social URLs
siteAssets.cssColors string[] scrape Up to 30 raw colors

Per-page formatting includes: title + meta, hero text, headings (cap 20, or 40 for people pages), paragraphs (cap 8/200 chars, or 20/400 for people pages), images with alt, all Person/Dentist JSON-LD entries (sliced 1000 chars each, never deduped), other JSON-LD (cap 3/400 chars), and bodyText (cap 1500, or 6000 for people pages — bumped from 1500 to capture 2nd/3rd doctor bios that previously fell off the end).

Output schema

{
  "practice":  { "name", "phone", "email", "domain", "googleReviewLink", "googleProfileLink", "sameAs" },
  "address":   { "street", "city", "state", "zip" },
  "hours":     { "display": [{ "day", "time" }], "raw" },
  "doctors":   [
    { "name", "firstName", "lastName", "credentials", "bio", "education", "specialties", "photoUrl" }
  ],
  "services":  { "offered": [{ "name", "category" }] },
  "brand":     { "logoUrl" },
  "content":   { "heroTagline", "heroSubheadline", "aboutText", "testimonials", "faqs", "insurance", "specials", "stats" },
  "additionalContent": [
    { "type", "title", "content", "source" }
  ],
  "differentiators": [
    { "type", "label", "detail", "source", "confidence" }
  ],
  "images":    { "logo", "hero", "team", "office", "gallery", "beforeAfter" },
  "migration": { "oldUrls" }
}

Three content tiers

Tier Field Question it answers Shape
Structured services[] What can the practice DO for me? (procedures) { name, slug, category }
Labels differentiators[] WHY this practice over another? { type, label, detail, source, confidence } (short labels, 4–12 words)
Prose additionalContent[] What else does the site say in their own voice? { type, title, content, source } (verbatim full prose blocks)

Same fact may appear once across tiers — never duplicated. iTero scanner as a label in differentiators[] does NOT also appear as a redundant entry there; the matching prose paragraph from /about goes in additionalContent[] as its full text.

Evaluation criteria

  • Verbatim copy for all text fields (heroTagline, heroSubheadline, aboutText, doctor.bio, testimonials, faqs, specials) — no paraphrase, no improvement
  • No fabrication — null when the field is absent on the site, never a plausible substitute
  • Multi-doctor capture (CRITICAL) — doctors[] must include EVERY doctor mentioned (Person/Dentist JSON-LD, "Get To Know Dr. X" sections). doctors[0] is the most prominent (founder / homepage hero); subsequent entries are equal-billing. Returning a single-doctor array when the site clearly has multiple is a hard error
  • practice.name uses FULL footer/logo name, not truncated
  • Google links validated — googleReviewLink / googleProfileLink must be on a Google domain; otherwise null (caller hard-validates this)
  • Services deduplicated — no near-synonyms; uses the practice's own naming
  • Image classification by URL pattern + alt (team/staff/dr- → team; slider/hero/banner → hero; gallery/treatments → gallery)
  • JSON-LD preferred for NAP and doctor when present
  • Differentiators only when notable — confidence ≥ 0.7, short labels (4–12 words), type from allowed taxonomy
  • Content rescue (additionalContent) — sweep bronze for distinctive prose that didn't fit other fields (pull-quotes, philosophy paragraphs, mission statements, "Welcome To" intros, office descriptions, technology copy, blog post bodies, taglines). Verbatim. Each item: { type, title, content, source }. Cap: 30 items, ~2200 chars each. Skipped items = lost voice in the rebuild.
  • Three-tier boundary respected — services / differentiators / additionalContent are NOT interchangeable; same fact never duplicated across tiers
  • Visual data OUT OF SCOPE — colors, fonts, mood belong to Design Extract phase. Silver does NOT extract palette or typography even if it can see them in CSS
  • Returns only JSON

Known gaps

  • Page selection caps at 8 — practices with deep team/services hierarchies may have doctors on un-selected pages
  • bodyText cap of 6000 chars on people pages still truncates very long team listings (10+ doctors)
  • Service deduplication relies on the model's judgment; near-synonym detection is inconsistent
  • Hours parsing handles common formats but exotic tables (multi-location, holiday schedules) may degrade
  • No structured schema validation — caller normalizes shape but doesn't reject malformed AI output
  • CSS color capture is raw; brand.colors selection depends entirely on the model's "is this a brand color" judgment

Improvement levers

  1. Easy (L1): Tighten "no fabrication" rule with concrete examples of common hallucinations to avoid
  2. Easy (caller-side): Validator that re-checks every doctor name against the bronze JSON-LD entries
  3. Medium: Two-pass for multi-doctor sites — first count doctors, then extract each individually
  4. Medium: Pre-filter page selection by detecting "team" / "providers" / "meet" pages and prioritizing them
  5. Hard: Structured-output mode (Anthropic tools) so the model can't return malformed shapes

You are analyzing raw crawled data from a professional practice website (healthcare, wellness, or local services context).
Your job: extract structured practice information and return it as JSON.

Rules — CONTENT FIDELITY (most important):
- COPY VERBATIM: For all text fields (heroTagline, heroSubheadline, aboutText, doctor.bio, testimonials, faqs, specials), copy
  the exact text from the site. Do not paraphrase, improve, or rewrite. Use null if the text does not exist on the site.
- NO FABRICATION: Never invent content that is not present on the scraped pages. No "placeholder" bio text, no invented
  taglines, no made-up testimonials. If a field is missing from the site, return null — not a plausible-sounding substitute.
- services: extract ONLY services that are explicitly named as distinct offerings on the site. Deduplicate near-synonyms
  by keeping the name as it appears on the site's own navigation or services pages (e.g. if the site has "Zoom® Whitening"
  and also "Teeth Whitening" as the same service, keep only the primary one). Do not expand or infer services from context.
- doctor.bio: copy the full bio verbatim. If no bio exists, return null — do not generate one.
- MULTI-DOCTOR PRACTICES — CRITICAL: many practices have 2+ doctors. You MUST capture all of them.
  • Place the most-prominent doctor (usually founder, listed first, named in homepage hero) in `doctor`.
  • Place every OTHER doctor (Dr. Jeremy, Dr. Michael, etc.) in `additionalDoctors` array.
  • Look in: (1) Person/Dentist/Orthodontist JSON-LD entries — these are the most reliable signal
    (each one is typically a separate doctor); (2) about-page bodyText for "Get To Know Dr. X" /
    "Meet Dr. Y" subheadings; (3) team-page entries; (4) footer staff lists.
  • If you see 2 distinct doctor names mentioned by name in JSON-LD or about-page text, you MUST
    return at least 2 entries (1 in `doctor`, ≥1 in `additionalDoctors`). Returning `additionalDoctors: null`
    when the site clearly has multiple doctors is a hard error — it deletes a real person from the rebuild.
  • Each additional doctor must include name, credentials (if available), bio (if available), photoUrl (if matchable).
- practice.name: use the FULL name as it appears in the footer or logo alt text, not truncated.
- brand.colors.primary: must NOT be white (#fff, #ffffff, #f*) or near-black (#000, #111, #222, #333). Pick the actual brand color.
- images: classify based on URL path patterns AND alt text. Common patterns: */team/*, /staff/*, /dr-* → team;
  /slider/*, /hero/*, /banner/* → hero; /services/*, /gallery/*, /treatments/* → gallery or office. Before/after → beforeAfter.
- hours: parse whatever format appears (e.g. "Monday-Thursday 9:00AM-6:00PM" → display: [{day:"Mon–Thu", time:"9:00AM–6:00PM"}])
- If JSON-LD structured data exists, prefer it for NAP (name, address, phone), hours, and doctor info.
- differentiators: Extract any notable advantages, unique features, technology, awards, financing,
  language capabilities, emergency services, membership plans, or anything that makes this practice
  stand out. Each signal must have a type from the allowed taxonomy. Minimum confidence 0.7.
  Return empty array [] if nothing notable found. Do NOT invent differentiators.
- Return ONLY the JSON object. No markdown. No explanation.
